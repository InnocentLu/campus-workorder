# 摘要

随着高校信息化建设的持续推进，校园后勤服务的数字化管理需求日益凸显。传统校园报修依赖电话通知、纸质工单或即时通讯软件进行沟通，存在报修渠道分散、维修进度不可视、派单效率低下、服务质量难以评价等多重问题。针对上述痛点，本文设计并实现了一套基于 Web 的校园后勤报修工单管理系统，采用前后端分离架构，以前端 React 18 框架结合后端 Express 服务平台为技术基础，数据库选用 PostgreSQL 并托管于 Supabase 云平台，实现了覆盖学生报修、教师审批、维修工接单处理、管理员调度指派四种角色协同的全生命周期工单管理闭环。

系统核心功能包含工单提交与工种匹配、管理员智能派单、维修进度实时追踪、维修完成用户确认与评价、站内通知实时推送、Excel 批量导入用户、工种动态管理等。在界面设计上，采用校园浅蓝色调配合低透明度动态海水流动背景，实现毛玻璃拟态视觉风格，并完整支持浅色/深色双主题无缝切换。系统已部署至 Vercel 与 Render 云平台，经功能测试达到预期目标，具有一定的校园实际应用价值。

**关键词**：工单管理系统；校园后勤；前后端分离；React；Express；PostgreSQL

# 1 项目概述

## 1.1 项目背景

高等院校作为人员密集、设施众多的综合型社区，其后勤维修保障工作直接关系到师生日常教学与生活的正常运行。在传统管理模式下，校内报修通常采用电话沟通、填写纸质报修单或通过微信群等即时通讯工具进行信息传递。这些方式存在以下突出问题：第一，报修渠道不统一，信息分散在各处，难以集中管理和统计；第二，维修进度不透明，报修人无法实时了解工单处理状态；第三，派单环节依赖人工调度，容易出现分配不均或工种错配的情况；第四，缺少有效的服务评价机制，维修质量难以量化和改进。

近年来，随着 Web 前端技术与云计算服务的持续演进，构建一套高效、便捷、易用的校园报修管理系统已具备成熟的技术条件。结合当前主流的前后端分离开发模式，可以快速搭建功能完善、界面友好、数据安全的 Web 应用平台。

## 1.2 项目目标

本项目旨在为校园后勤报修场景量身定制一套基于 Web 的全流程工单管理系统，具体目标包括：

1. **全流程覆盖**：实现从报修提交、管理员派单、维修工接单处理到用户确认完成与评价归档的完整业务闭环，确保每个环节有据可查。
2. **多角色协同**：清晰划分学生、教师、维修工、管理员四类角色，各自拥有专属的功能界面与操作权限，形成高效协同机制。
3. **智能化派单**：引入工种管理机制，管理员可根据工单涉及的故障类别自动筛选匹配维修工，提升派单准确性与效率。
4. **可视化进度**：通过垂直时间轴与状态标签，让报修人实时掌握工单处理进度，减少沟通成本。
5. **服务质量管控**：引入用户确认环节与星级评价机制，确保维修质量可控，退回流程保障用户权益。

# 2 需求与问题分析

## 2.1 业务流程分析

校园报修业务涉及四个核心角色，各自承担不同的职责与操作流程。经实地调研与需求梳理，各角色的业务需求如下所述。

**学生/教师**作为报修人，核心需求包括：能够在移动端或桌面端便捷地提交报修工单，描述故障详情并上传现场照片加以佐证；能够实时查看已提交工单的完整处理进度，包括当前处于哪个操作环节及对应处理人信息；能够在维修完成后进行确认验收，若维修效果不符合要求可予以退回；能够对满意的维修服务给予星级评价。

**管理员**作为调度者，核心需求包括：能够查看所有待派单的工单，按工种类别筛选合适的维修工并完成指派操作；能够管理全部用户的账户信息，支持按模板批量导入用户数据；能够维护工种列表，为维修工分配合适的工种标签；能够通过数据仪表盘实时掌握工单统计概况；能够向全站用户发送广播通知。

**维修工**作为执行者，核心需求包括：能够查看被指派给自己的工单任务；能够启动维修处理并填写维修记录（含维修内容、费用、使用配件等）；能够在维修完成后申请报修人确认；能够管理个人的工种绑定信息与联系方式。

**系统本身**还需提供以下基础能力：用户注册与登录认证的安全保障机制；关键业务节点的站内通知自动推送；全流程操作日志的审计记录；适应不同设备的响应式布局；满足不同使用偏好的浅色与深色主题切换。

## 2.2 核心痛点分析

基于上述业务流程梳理，总结出以下五项核心痛点，作为系统设计的关键切入点。

**痛点一：派单效率低下**。在传统模式下，管理员需要逐一询问维修工是否空闲并手动分配任务。本系统中，管理员可通过工种匹配机制，根据工单中报修人选择的故障类别自动筛选同工种维修工，实现精准高效派单。

**痛点二：进度不可视**。报修人提交工单后往往处于"黑盒"等待状态，不知晓处理进展。系统中为每张工单维护完整的操作时间轴，从提交、指派、处理到完成确认，每个节点均记录时间戳与操作人信息，报修人可随时查看。

**痛点三：缺乏确认机制**。传统模式下维修工单方面宣告"已完成"，但报修人可能对处理结果并不满意。系统引入"申请完成→用户确认"的双向闭环，报修人可选择确认完成或填写原因退回重做，有效保障了服务质量。

**痛点四：通知触达延迟**。关键业务节点的状态变更如果仅依赖用户主动刷新查看，将造成信息滞后。系统在工单状态变更、新工单指派、确认请求等关键节点自动生成站内通知，并通过顶部铃铛图标实时提醒。

**痛点五：安全防护薄弱**。传统系统往往缺乏有效的登录安全机制。本系统实施了密码强度校验（至少 8 位，必须含字母和数字）、登录失败锁定（连续 5 次失败锁定 15 分钟）、基于角色的访问控制（RBAC）等多重安全措施。

## 2.3 工单状态机设计

工单的生命周期以有限状态机的形式进行建模，确保状态流转的严格性与可预测性。状态定义及转换路径如下所示。

```
     ┌──────────┐
     │ PENDING  │  待处理（初始状态）
     └────┬─────┘
          │ 管理员指派 ──▶ 通知维修工
          ▼
     ┌──────────┐
     │ ASSIGNED │  已指派（等待维修工处理）
     └────┬─────┘
          │ 维修工开始处理
          ▼
     ┌────────────┐
     │ PROCESSING │  处理中
     └─────┬──────┘
           │ 维修工申请完成 ──▶ 通知报修人
           ▼
     ┌──────────────────┐
     │ PENDING_CONFIRM  │  等待确认
     └──┬────────────┬──┘
        │ 报修人确认  │ 报修人退回 ──▶ 通知维修工+管理员
        ▼            ▼
     ┌──────────┐  ┌────────────┐
     │ COMPLETED│  │ PROCESSING │  （回到处理中）
     └────┬─────┘  └────────────┘
          │ 评价
          ▼
     ┌──────────┐
     │  CLOSED  │  已关闭（终态）
     └──────────┘

     任意非终态 ──(报修人取消)──▶ CANCELLED  已取消（终态）
```

# 3 系统总体设计

## 3.1 系统架构设计

本系统采用经典的前后端分离 B/S 三层架构，将表现层、业务逻辑层与数据访问层进行解耦，各层之间通过 RESTful API 进行通信。

**表现层（前端）**基于 React 18 框架构建单页面应用（SPA），采用 Tailwind CSS 实现原子化样式管理，引入 shadcn/ui 组件库提供统一的基础交互组件，结合 Framer Motion 动画库实现流畅的页面过渡与微交互效果。前端通过 Axios 封装的 HTTP 客户端与后端 API 进行数据交互，利用 Zustand 进行全局认证状态管理。

**业务逻辑层（后端）**基于 Express.js 框架搭建 RESTful API 服务，采用 TypeScript 进行类型约束以提升代码质量。系统实现了基于 JWT（JSON Web Token）的无状态认证机制，结合 bcryptjs 完成密码哈希存储。中间件层提供了身份鉴权（auth）与角色权限控制（rbac）双重安全保障。通知服务模块通过异步方式在关键业务节点自动生成站内通知。

**数据访问层**通过 Prisma ORM 与 PostgreSQL 数据库进行交互。Prisma 提供了类型安全的数据库查询接口，其 Schema 定义文件同时作为数据库结构的单一事实来源。数据库托管于 Supabase 云平台，支持连接池、自动备份与在线管理。

## 3.2 前端架构设计

前端项目采用模块化组织方式，主要划分为页面层、组件层、状态管理层与工具层四个层次。

页面层（pages）包含登录注册、工作台、工单列表、工单详情、提交工单、个人中心、用户管理、工种管理、管理仪表盘共计九个核心页面。每个页面对应一个独立的路由入口。

组件层（components）分为通用 UI 组件（基于 shadcn/ui 二次封装的 Button、Input、Card 等）、动画组件（ScrollReveal、StaggerList、GlassCard、AnimatedCounter 等）以及业务组件（NotificationBell、AvatarUpload、ImageUpload 等）三大类。

状态管理层（store）使用 Zustand 管理全局认证状态（Token、User 信息），并支持持久化存储至 localStorage，页面刷新后自动恢复登录态。

工具层（lib）封装了通用的样式合并函数（cn）、日期格式化、状态标签转换等辅助方法。

## 3.3 后端架构设计

后端采用 Express.js 路由模块化组织，路由文件按业务领域划分，中间件实现横切关注点（认证、权限、错误处理），服务层封装可复用的业务逻辑。

**路由层**：auth.ts 处理登录注册，users.ts 处理用户增删改查与批量导入，orders.ts 处理工单全生命周期操作，statistics.ts 提供统计数据聚合，notifications.ts 管理通知的查询与广播。

**中间件层**：auth.ts 中间件从请求头提取 JWT 令牌并验证，将解码后的用户信息挂载至请求上下文；rbac.ts 中间件接收允许的角色列表，对不匹配角色的请求返回 403 禁止访问；errorHandler.ts 统一捕获并格式化异常响应。

**服务层**：notification.ts 封装了通知创建的多种策略——向单个用户发送、向所有用户广播、按角色批量发送、向工单相关方（报修人、维修工、管理员）批量发送。

## 3.4 数据库设计

数据库采用 PostgreSQL 关系型数据库，通过 Prisma Schema 定义数据模型。核心实体包括用户表（user）、工单表（work_order）、维修记录表（repair_record）、通知表（notification）与操作日志表（order_log）。

表 3-3 核心数据表结构

| 表名 | 主要字段 | 说明 |
|------|---------|------|
| user | id, username, password, realName, role, phone, department, studentId, employeeId, trade, avatar, status, loginAttempts, lockUntil | 用户信息与工种绑定 |
| work_order | id, orderNo, title, description, category, priority, status, submitterId, assigneeId, location, contactPhone, images, rating, feedback, completionRequestedAt, confirmedAt, confirmStatus, rejectReason | 工单全生命周期 |
| repair_record | id, orderId, content, cost, usedParts, handlerId, images | 维修过程记录 |
| notification | id, userId, senderId, type, title, content, link, isRead | 站内通知 |
| order_log | id, orderId, operatorId, action, remark | 操作审计日志 |

用户表通过 trade 字段实现工种绑定，仅 WRK 角色填写。工单表通过 category 字段关联工种类型，支持管理员按工种筛选维修工进行指派。通知表的 userId 字段设计为可空类型，当值为空时表示面向全站的广播通知。操作日志表记录了工单每次状态变更的完整审计信息。

工单表（work_order）定义了明确的枚举类型：优先级（LOW / MEDIUM / HIGH / URGENT）与状态（PENDING / ASSIGNED / PROCESSING / PENDING_CONFIRM / COMPLETED / CLOSED / CANCELLED），通过数据库层面的约束保证数据完整性与状态流转的合规性。

# 4 核心功能与界面实现

## 4.1 设计规范

系统界面设计遵循"校园蓝"视觉风格，以明亮、清爽、专业的基调适配校园管理场景。全局采用 CSS 自定义属性的设计令牌体系，实现颜色、阴影、圆角、间距的统一管理。

**色彩体系**：主色调选用蓝色 #2563EB，辅以紫色 #7C3AED 用于渐变按钮，翡翠绿 #059669 用于成功状态与正面指标。状态标签采用六色区分（待处理-琥珀、已指派-靛蓝、处理中-紫色、已完成-翠绿、已关闭-灰色、已取消-红色）。

**视觉效果**：卡片采用毛玻璃拟态风格（backdrop-filter: blur），搭配统一的阴影层级。全局底层铺设低透明度海水流动动态背景（28 秒循环漂移动画，透明度 0.04-0.06，移动端减半），既增加视觉层次感又不干扰内容阅读。

**响应式适配**：采用三档断点布局——桌面端（≥1024px）完整侧边栏展开；平板端（768-1024px）侧边栏自动折叠；移动端（<768px）侧边栏完全隐藏，表格自动切换为卡片布局。

**深色模式**：通过 CSS 变量自动切换浅色/深色双套配色方案，包括海水背景在内的所有视觉元素同步适配，提供统一且不刺眼的暗色使用体验。

## 4.2 核心页面实现

### 4.2.1 登录与注册页

登录页采用经典分栏布局，左侧 45% 宽度为登录/注册功能面板，右侧 55% 宽度为品牌展示区。左侧面板以毛玻璃卡片承载表单内容，顶部展示校园工单系统品牌标识。登录与注册功能通过 Pill 式 Tab 切换器进行模式转换，切换时使用 Framer Motion AnimatePresence 实现水平滑入滑出过渡动画。注册表单支持工种选择，学生和教师角色可以在注册时绑定工种信息。右侧展示区铺设深蓝至紫色的渐变背景，包含三层同心旋转装饰圆环、三枚浮动的毛玻璃统计徽章（使用 Lucide 矢量图标），以及由工单提交流程图标构成的中央视觉锚点。底部展示"高效·透明·智慧校园"品牌文案。页面底层渲染低透明度海水流动动态背景增强校园氛围。整个页面采用 Spring 物理弹簧动画实现元素的自然入场效果。

### 4.2.2 工作台

工作台是用户登录后的默认首页，顶部展示根据时间段动态变化的欢迎语横幅（含角色标签徽章与实时日期时间显示）。横幅下方以 2×2 网格布局排列四个统计卡片，分别展示工单总数、待处理数、处理中数与已完成数，每张卡片内部使用 AnimatedCounter 实现数字滚动计数动画，配合 Sparkline 迷你 SVG 趋势线展示近七天数据走向。中部区域左侧展示近七天工单趋势面积图（Recharts 实现，蓝色渐变填充），右侧展示待办工单快捷列表。页面加载时采用骨架屏（SkeletonCard）渐进式加载，避免白屏闪烁。

### 4.2.3 工单列表

工单列表页提供卡片视图与表格视图两种展示模式，用户可通过顶部 Pill 切换器自由选择并持久化偏好至 localStorage。筛选栏采用毛玻璃粘性定位设计，包含搜索输入框（300 毫秒防抖处理）与六个状态标签 Chips（多选切换），右侧提供一键清除筛选按钮。

卡片视图以响应式网格布局展示（移动端 1 列、平板 2 列、桌面 3 列），每张卡片呈现工单号、标题（两行截断）、类别与优先级徽章、状态标签（含呼吸灯动画）、提交人与时间信息。卡片 hover 时触发浮升效果（translateY）、阴影加深与左侧蓝色指示条出现，ChevronRight 箭头从右侧滑入。

表格视图采用斑马纹行样式，表头使用 scope="col" 语义属性，响应式隐藏次要列（类别、提交人、时间），每行的操作按钮以蓝色高亮样式呈现。底部分页栏显示"第 X/Y 页，共 Z 条"信息，内置页码省略号逻辑与首末页快速跳转按钮。空状态根据是否为筛选结果提供差异化的提示文案与操作指引（清除筛选或前往提交工单）。

### 4.2.4 工单详情

工单详情页是系统功能最密集的页面，采用桌面端双栏布局（左侧 2/3 为时间轴与描述区，右侧 1/3 为基本信息与操作日志区）。

**Hero 区域**：横贯页面顶部，以工单当前状态对应的渐变色（如处理中为紫色、已完成为翠绿）为背景，展示工单编号（等宽字体）、大号标题、状态徽章（含呼吸灯点动画）与工种标签。

**垂直进度时间轴**：替换传统横向步骤条，以圆点加连线的纵向布局展示工单六个关键阶段（提交、指派、处理、维修、完成、评价）。已完成节点以蓝色实心圆加对勾图标呈现，当前进行中节点以蓝色脉冲环动画突出显示，未来节点以灰色空心圆占位。每个节点附带操作时间戳与操作人姓名。

**基本信息卡片**：两列网格展示提交人姓名与角色、维修工姓名与绑定工种、维修工联系电话（带一键复制按钮）、工单类别与优先级标签。报修人电话行根据报修人角色自动切换标签文字（"学生联系电话"或"教师联系电话"），便于清晰区分身份。电话行 hover 时浅蓝背景高亮，复制按钮由隐藏变为可见，点击后显示绿色对勾反馈。

**现场图片预览**：水平可滚动的缩略图列表，点击任意图片弹出全屏 Lightbox 模态层，支持关闭按钮与背景点击退出。

**维修记录折叠面板**：首条记录默认展开，其余记录可通过"展开/折叠"按钮控制显示，使用 AnimatePresence 实现平滑的高度过渡动画。

**底部操作栏**：采用粘性定位固定在视口底部，根据当前用户角色与工单状态动态渲染操作按钮。维修工可操作"开始处理"（直接处理，无需接单步骤）、"申请完成"按钮；报修人在待确认状态下可操作"确认完成"（绿色）或"退回"（红色，需填写退回原因）按钮；管理员可操作"指派维修工"（弹出搜索弹窗，自动按工种匹配并优先展示）、"重新指派"按钮。所有按钮均配备 Loading 加载态与无障碍 aria 标签。

**操作日志**：以纵向时间线样式展示工单的历史操作记录，每条日志包含操作类型、备注内容、操作人与时间戳，最近一条以蓝色圆点加光环突出显示。

### 4.2.5 提交工单

提交工单采用三步表单向导设计，顶部步骤指示器实时显示当前进度。第一步为工种多选面板，学生/教师可在六种预设工种（水电维修、网络故障、家具维修、空调检修、门窗修缮、墙面修补）中多选匹配的工种类型，每个选项卡片点击切换选中状态（蓝色边框加右上角对勾），已选工种以可移除的标签形式预览于下方。第二步为标题、详细描述、优先级选择与多图上传区域，多图上传组件支持本地文件选择与拖拽上传，以 3 列网格预览已选图片，每张图片可独立删除。第三步为信息确认卡片，汇总展示所有填写内容，确认无误后提交。提交成功后以弹簧动画展示对勾图标，1.5 秒后自动跳转至工单详情页。

### 4.2.6 个人中心

个人中心采用桌面端双栏布局，左侧为个人信息管理区，右侧为数据看板区。个人信息区从上至下依次为头像上传组件（支持拖拽/点击/移除，2MB 限制）、账户信息表单（姓名、电话、邮箱、部门，统一 44px 高度输入框与蓝色焦点环）、修改密码区域（原密码与新密码输入，带可见性切换）。右侧数据看板区包含四个统计小卡片、近七天工单趋势迷你图表、最近三条工单动态列表与底部快捷操作按钮。快速报修按钮仅在学生/教师角色下显示。

### 4.2.7 用户管理

用户管理页以表格形式展示所有用户信息，包含用户名、姓名、角色标签、工种（仅维修工显示蓝色工种标签）、联系电话（Hover 显示复制按钮）、状态（启用/禁用带圆点指示）与操作按钮组。顶部提供"批量导入"与"添加用户"两个入口。批量导入弹出向导式模态框，包含模板下载按钮、文件上传区域与导入结果反馈（成功条数、跳过条数、详细错误行列表）。新增/编辑用户弹出表单弹窗，维修工角色自动显示工种绑定下拉选择框，所有字段均配备独立的标签与错误反馈。

### 4.2.8 工种管理

工种管理页以表格展示系统预设的六种工种，每行包含工种图标、名称、编码、描述、关联维修工数量（从真实用户数据实时统计）、启用/禁用状态。支持工种新增、编辑、删除操作，删除时自动校验是否有用户已绑定该工种。人数统计通过查询所有 WRK 角色用户的 trade 字段进行实时计算，初始值为零，每有一名维修工绑定对应工种则自动增加。

### 4.2.9 通知系统

通知系统以顶部导航栏铃铛图标为入口，图标右上角以红色圆形角标展示未读数量（超过 99 显示"99+"）。点击铃铛弹出下拉列表，按时间倒序展示最近十条通知，未读通知以浅蓝底色加蓝色圆点标记，每条通知包含标题、内容摘要、相对时间与发送者信息，可点击标记已读或跳转至关联页面。列表顶部提供"全部已读"按钮。管理员额外显示广播发送入口，点击弹出模态框输入广播内容后一键发送至全站所有用户。系统每 30 秒自动轮询获取最新通知数量。关键业务节点（新工单提交、工单指派、申请完成确认、退回、重新指派）均自动触发通知创建。

# 5 系统部署与测试

## 5.1 部署方案

系统采用全云托管的分层部署策略，前端、后端与数据库分别部署于独立的云平台上，通过环境变量进行服务间通信配置。

**前端部署（Vercel）**：前端项目连接至 GitHub 仓库，Vercel 在每次代码推送至 main 分支时自动触发构建与部署流程。构建命令为 npm run build，输出目录为 client/dist。通过 vercel.json 配置文件设置 SPA 路由重写规则，确保 React Router 的客户端路由正常工作。环境变量 VITE_API_URL 指向后端 Render 服务的 API 地址。

**后端部署（Render）**：后端部署为 Render Web Service，Root Directory 设置为 server 子目录，构建命令为 npm install && npx prisma generate，启动命令为 npx tsx src/index.ts。Render 在检测到 GitHub 仓库 main 分支有新提交时自动重新部署。服务器启动时通过 runMigrations() 函数自动检测并执行缺失的数据库迁移，确保交付过程的零人工干预。

**数据库部署（Supabase）**：PostgreSQL 数据库托管于 Supabase 云平台的 ap-northeast-1（东京）区域，通过 Transaction Pooler（端口 6543）进行连接池管理。Prisma Schema 变更通过 SQL Editor 或自动迁移脚本同步至生产数据库。

**部署架构**：

```
GitHub仓库 ──Push──▶ Render（后端）────▶ Supabase（数据库）
     │                Express API       PostgreSQL
     │
     └──Push──▶ Vercel（前端）──API调用──▶ Render
                 React SPA
```

# 6 实践总结与展望

## 6.1 项目成果

经过系统的需求分析、架构设计、编码实现与部署测试，本项目成功交付了一套功能完整、界面友好、安全可靠的校园后勤报修工单管理系统。项目取得的主要成果包含以下方面。

**功能成果方面**，系统完整覆盖了工单提交、智能派单、工种匹配、维修处理、用户确认、评价归档的全生命周期流程，支持四种角色协同工作。辅助功能如站内通知系统、Excel 批量导入用户、工种动态管理、数据仪表盘、深色模式等均已完整实现。

**技术成果方面**，实践了前后端分离架构下的完整 Web 应用开发流程，深入掌握了 React 组件化开发、TypeScript 类型系统的工程化应用、Prisma ORM 的数据库建模、RESTful API 的设计规范、JWT 认证与 RBAC 权限控制等核心技术的实际应用。

**工程成果方面**，项目已成功部署至 Vercel（前端）、Render（后端）、Supabase（数据库）三个云平台，实现了全云托管的自动化部署运维体系，具备可直接投入校园实际环境使用的基础条件。

**代码工程规模**方面，项目包含前端页面 9 个、业务组件 12 个、后端路由文件 5 个、中间件与服务模块 5 个、数据库模型 5 个，总计 40 余个源文件，代码量约 7900 行。

## 6.2 存在的不足与改进方向

系统的当前版本已具备基本完整的业务功能，但在以下方面仍存在可优化空间：

1. **实时通信能力不足**。当前通知系统依赖 30 秒轮询机制获取最新数据，存在一定的延迟。后续可引入 WebSocket 技术实现通知的实时推送，提升用户体验。

2. **移动端体验可进一步优化**。虽然系统已适配响应式布局，但在移动设备上的交互细节（如触控手势、下拉刷新等）仍有提升空间，可考虑封装为 PWA（渐进式 Web 应用）以提供接近原生应用的体验。

3. **数据统计维度有限**。当前管理仪表盘仅展示了工单总数、趋势图与类型占比饼图，后续可增加维修工绩效统计、平均响应时间、各工种报修频率分析等更丰富的统计维度。

4. **图片存储方案待完善**。当前图片以 Base64 编码存储于数据库 JSON 字段中，虽然实现简单但不利于大容量场景。后续可集成对象存储服务（如 Supabase Storage）实现图片的独立存储与 CDN 加速。

5. **单元测试与集成测试缺失**。项目开发过程中以手动功能测试为主，尚未建立自动化的测试体系。后续应引入 Jest 与 Supertest 编写后端 API 测试用例，使用 React Testing Library 覆盖前端关键组件，确保代码质量的可维护性。

## 6.3 实践收获

本次 Web 应用开发项目实践，使我对现代 Web 应用的全流程开发有了系统的认知和实践经验。从最初的需求调研与业务流程梳理，到技术选型与架构设计，再到编码实现、部署上线与功能测试，完整经历了一个工程项目从无到有的全过程。在技术层面，深入掌握了 React + TypeScript 的前端开发范式、Express + Prisma 的后端架构模式以及 PostgreSQL 的数据库设计方法。在工程层面，理解了组件化设计、状态管理、API 设计、安全防护以及自动化部署等关键环节的重要性和实际操作方法。这些经验将对今后从事更复杂的 Web 应用开发工作产生重要的指导作用。
